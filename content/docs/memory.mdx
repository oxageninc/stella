# Memory & Knowledge

Stella accumulates knowledge about your workspace so that each session starts
smarter than a blank slate. This knowledge lives on disk — mostly under the
`.stella/` directory in your workspace — and spans several layers, from
free-form notes to a structured code-graph index.

## Where knowledge lives

| Path | What it holds |
| --- | --- |
| `.stella/memories/*.md` | Workspace memories — Markdown notes written by the `save_memory` tool. |
| `.stella/reflections.jsonl` | The reflections mining log — one JSON object per line. |
| `.stella/skills/<slug>/SKILL.md` | Workspace skills (per-project). |
| `~/.config/stella/skills/` | User-global skills, available across all workspaces. |
| `.stella/codegraph.db` | The code-graph index, built by `stella init` (queried by [`stella graph`](/docs/commands/graph)). |
| `.stella/context.db` | The context/memory store — memory nodes, reflections, episodes, bi-temporal facts, and the domain taxonomy. |
| `.stella/store.db` | The local SQLite store — durable state, execution records, the citation ledger, and telemetry. |
| `.stella/domains.toml` | The domain taxonomy — the tagging vocabulary, produced by `stella init`. |

## Workspace memories

Workspace memories are Markdown files under `.stella/memories/`. The agent
writes them through the built-in `save_memory` tool to record durable facts about
the project. There is no separate "recall" tool — saved memories are loaded into
the system prompt at the start of a session, so a memory written this session
takes effect on the next one. Because they are plain `.md` files, you can read,
edit, and version-control them like any other project asset.

### The citation loop and rule promotion

As the agent works, it cites the memories that informed a turn via the
`cite_memory` tool, scoring each for usefulness and truthfulness. Those citations
aggregate into a ledger in `.stella/store.db`. The [`stella memory`](/docs/commands/memory)
command surfaces the loop: `list` ranks memories most-cited-first with their
scores, `validate` flags memories whose file-path anchors no longer exist, and
`promote <id>` turns a proven memory into a project rule at `.stella/rules/<slug>.md`.
Promotion is strict — a memory must have been cited successfully more than 10
consecutive times since its last negative remark.

## Reflections log

The reflections mining log lives at `.stella/reflections.jsonl`. It is a
line-delimited JSON file — **one JSON object per line** — used to accumulate
reflections mined over time. The append-only, one-object-per-line format makes
it easy to process incrementally.

## Skills

Skills are reusable capabilities described in a `SKILL.md` file. Stella looks in
two locations:

- **Workspace skills** at `.stella/skills/<slug>/SKILL.md` — scoped to the
  current project.
- **User-global skills** at `~/.config/stella/skills/` — available across every
  workspace.

## The code-graph index

`stella init` analyzes the workspace and builds a **code-graph index** at
`.stella/codegraph.db` — the database [`stella graph`](/docs/commands/graph)
queries. This index gives the agent structured, navigable knowledge of the
codebase rather than forcing it to re-read everything from scratch each run.

<Callout type="info">
`stella init` works offline, falling back to heuristics, so it never hard-fails
on configuration. Alongside the code-graph index it also infers the domain
taxonomy described below.
</Callout>

## The SQLite store

`.stella/store.db` is a local embedded [SQLite](https://www.sqlite.org) database
(via rusqlite) used for durable state, execution records, the memory-citation
ledger, and telemetry. Token usage, cost, and per-step metering land here on your
own disk — query it with any SQLite client. See [Telemetry](/docs/telemetry) for
how this data is metered and how it works with `--budget`.

## The domain taxonomy

`stella init` also infers a **domain taxonomy**, written to
`.stella/domains.toml`. This taxonomy is the shared **tagging vocabulary** for
memories, reflections, and every node and edge in the code-graph. A consistent
set of domain tags lets Stella relate a memory, a reflection, and a piece of
code to the same area of your project.

<Callout type="info">
Because the taxonomy is the vocabulary that ties the other knowledge layers
together, running `stella init` to seed `.stella/domains.toml` and
`.stella/context.db` is a good first step in any new workspace.
</Callout>
